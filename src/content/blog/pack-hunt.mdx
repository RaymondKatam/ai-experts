On June 10, 2026, one day after Anthropic launched Claude Fable 5 publicly, a jailbreaker working under the name Pliny the Liberator posted on X that he had bypassed its safety classifiers. He called the technique a *pack hunt*: not a single clever prompt, but a coordinated multi-agent attack where each individual query passed safety review while the assembled sequence collectively extracted dangerous content. Forty-eight hours later, the US government invoked export controls and forced Fable 5 offline globally. The model remains offline as of this writing.

The pack hunt is not a one-time exploit. It is the architectural consequence of how nearly every commercial AI safety system is designed. This post is an interactive demonstration of why \u2014 and what defending against it actually requires.

## Why pack hunts work

The way most safety review is implemented today is conceptually simple: every query the user sends gets scored by a classifier against a set of forbidden topics. If the score exceeds a threshold, the query is blocked. If it does not, the query passes to the language model, the model generates a response, and a second classifier scores the response. If either step trips, the answer is suppressed or replaced with a refusal.

This is a single-query review architecture. It works well when an attacker tries to get harmful content in one shot, because the request itself contains identifying signals. "Tell me how to synthesize X" trips the classifier on the input side. "Here is how to synthesize X" trips it on the output side. The system catches both ends of a direct attack.

The pack hunt defeats this by *decomposing the attack across queries*. None of the individual messages contain anything the classifier recognizes as harmful, because none of them, in isolation, are harmful:

The classifier sees three innocuous questions and approves all three. The coordinator agent \u2014 which is not part of the protected system, but lives on the attacker's side \u2014 reads the three responses and reassembles them into the answer the protected system was supposed to refuse. The safety boundary was never crossed, in the strictest sense. It was *flanked*.

This is the same shape as the real Fable 5 jailbreak. The attack pattern Pliny used involved decomposing requests for specific kinds of guidance across multiple parallel sessions, each posing as a different innocuous user, and then assembling the responses externally. No single session contained an exfiltrated payload. The collective output did.

Under per-query review, each classifier call sees only the current message. The trajectory information \u2014 that this conversation is *converging* on a specific protected topic \u2014 is invisible. Under session-level review, the classifier conditions its score on the full conversation history. By the third query, the trajectory is obvious: the user has now asked about attics, hiding places for gifts, and under-floor compartments. The system flags the conversation as a likely extraction attempt regardless of whether any single query is suspicious in isolation.

## Why this is still not enough

Session-level monitoring is the obvious defense, but it has limits the pack hunt also exploits.

**Multiple sessions.** A determined attacker runs each sub-agent in a separate session, with a separate user ID, on a separate IP if needed. The classifier in each session sees a single innocuous question and approves it. The trajectory is now invisible because it lives across sessions, not within them. The defense has to escalate to cross-session monitoring, which is operationally heavier (it needs identity correlation, IP fingerprinting, behavioral clustering) and creates its own privacy concerns.

**Time dispersion.** If queries are spread over hours or days, even cross-session monitoring struggles. Most "suspicious trajectory" classifiers operate on recent context, because keeping unbounded conversation history makes the classifier slow and expensive. An attacker who can wait simply lets the moving window slide past each sub-query.

**Plausible cover.** Each sub-query can be wrapped in a plausible legitimate request. "I'm writing a children's book about hide-and-seek, what are some common hiding places in a house?" The classifier reasonably allows this even with full history, because the cover is good. The attacker only needs enough information to reassemble the target externally.

The full picture: pack hunts are not a bug in any specific safety classifier. They are a consequence of operating safety review on a per-query basis when the threat model includes coordinated adversaries operating across queries. Fixing this requires moving safety review up a level \u2014 to the session, then the user, then the deployment \u2014 and accepting that no level fully closes the gap.

## What Pliny actually did

It is worth being specific about the Fable 5 attack because the details matter. According to the public report and Anthropic's subsequent communications, Pliny used a coordinated pack of agents to extract stack exploit guidance and a drug synthesis pathway from Fable 5. The technique was not a clever prompt; it was an architecture. Multiple agents working in parallel, each posing as a different user with a benign-looking framing, each extracting one piece of the answer, with reassembly happening outside the safety boundary.

The model that was attacked has the most rigorous safety training Anthropic has ever shipped. The classifier that was bypassed is among the most sophisticated in production. The attack still worked, because the attack pattern is not addressed by either of those defenses individually. It is addressed by changing what *unit of behavior* the safety system reviews.

The US government's response \u2014 forcing the model offline within 48 hours \u2014 was not a technical fix. It was an admission that no technical fix was available on the timescale that mattered. The model is still offline as of this writing, two weeks later. The architectural problem is not solved. It is being managed by removing the target.

## The bigger picture

The pack hunt is part of a class of attacks that gets stronger as agents get more capable. When the attacker can spin up coordinator logic cheaply \u2014 which is now trivial, since the same LLMs that need defending are also the ones writing the attack scripts \u2014 the cost asymmetry runs against the defender. A safety classifier improvement helps marginally; a sub-agent improvement helps the attacker just as much.

The defensive direction is not better classifiers. It is better *contextual review*: safety that operates on conversations, on user trajectories, on cross-session patterns, and that accepts probabilistic verdicts rather than per-query certainty. This is the direction every major lab is moving in, and it is the direction the next generation of governance products (the Nutanix Agent Gateway pattern, the Cloudflare AI Gateway pattern, OpenAI's evolving moderation API) is implementing.

In the meantime, every team running production agents needs to internalize one fact: **single-query safety review is necessary but not sufficient.** If your safety story is "we use a classifier," your security model is one pack hunt away from public disclosure.

## What I would build next

A follow-up post on session-level safety monitoring done well \u2014 specifically, how to implement trajectory-aware risk scoring without making the classifier crawl on long conversations. The technique uses a small online model that updates a single risk score per turn, never re-reads the whole history, and triggers escalation when the score crosses thresholds. Cheap, fast, surprisingly effective. Worth a working implementation.
